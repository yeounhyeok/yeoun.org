Prologue: 인프라 개편의 이유</h3><p>기존 홈서버(N4000) 단일 노드 구성은 한계가 명확했다. <strong>집 IP가 외부에 노출되는 보안 위협</strong>과 N4000의 부족한 컴퓨팅 파워로 인한 VPN 처리 속도 저하가 주된 문제였다.</p><p>특히 이 구성으로 바꾸는 동기가 되었던 것은 다음과 같다. 개인 pc로 오버워치를 하던 중, 평소 15ms정도 나오던 핑이 기본 20ms부터, 주기적으로 50ms까지 오르며 캐릭터들의 움직임이 계속해서 튀기 시작했다.</p><p>이는 집 사설망 기기들이 Adguard home을 통한 Split DNS를 사용하는 도중에 해당 노드의 서비스가 너무 많아 Load가 지나치게 커지면서 생기는 Packet loss로 진단되었다. 따라서 더 큰 컴퓨팅 노드, 혹은 이 글에서 다룰 분산 컴퓨팅 아키텍쳐를 필요로 하였다.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://yeoun.org/content/images/2026/02/image-1.png" class="kg-image" alt="" loading="lazy" width="2000" height="1064"><figcaption><span style="white-space: pre-wrap;">Idle 상태의 top. 이미 도커 데몬에 터지기 직전이다.</span></figcaption></figure><p>이를 해결하기 위해 인프라를 확장하고 역할을 분리하기로 결정했다.</p><ol><li><strong>Identity &amp; Network Hub</strong>: OCI ARM 인스턴스로 이관 (보안 및 대역폭 확보)</li><li><strong>Additional Worker Node</strong>: 유휴 자원인 LG 노트북(N4200)을 추가 리눅스 서버로 전환</li></ol><p>이 글은 그 과정에서의 트러블 슈팅과 최종적으로 완성된 <strong>Dual-Path Network Architecture</strong>에 대한 기록이다.</p><hr><h3 id="1-the-huddle-%EC%9C%88%EB%8F%84%EC%9A%B0%EB%A7%8C-%ED%8E%B8%EC%95%A0%ED%95%98%EB%8A%94-lg-%EB%85%B8%ED%8A%B8%EB%B6%81%EC%97%90-%EC%9A%B0%EB%B6%84%ED%88%AC-%EC%8B%AC%ED%8F%90%EC%86%8C%EC%83%9D%EC%88%A0">1. 맞추어 우분투 efi를 복사해 주었고, 성공적으로 인식되었다.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://yeoun.org/content/images/2026/02/image.png" class="kg-image" alt="" loading="lazy" Network Migration: OCI ARM 노드를 활용한 VPN 허브 구축</h3><p>기존에는 OCI amd 노드를 public entrypoint로 하기 위한 vps에서 FRP를 거쳐 집 내부의 VPN 서버로 접속하는 것으로 구상했으나, FRP 터널링 오버헤드로 인해 속도 저하가 심각했다. (300Mbps -&gt; 30Mbps) </p><p>개인적인 관리용 네트워크(Private)는 굳이 FRP를 거칠 필요가 없었고, 나는 항상 보안과 편의를 위해 VPN을 켜놓고 다니기 때문에 OCI ARM 노드에 클라이언트의 wireguard 터널을 직접 연결하기로 했다.</p><p>무조건 묻고 답하는 것이 한 세트인 TCP와 달리 UDP는 키가 안 맞을 시 응답조차 안하기 때문에 간단한 스캐너들에 대한 포트노출을 최소화 하기에 충분하다고 생각했다.</p><h4 id="21-revised-topology">2.1. Revised Topology</h4><ul><li><strong>기존:</strong> User / Admin -&gt; OCI VPS Entry (FRP) -&gt; Home N4000 (VPN Hub) <ul><li> Slow &amp; IP Exposed ❌</li></ul></li><li><strong>변경:</strong> <ul><li>Admin -&gt; OCI ARM (Direct UDP) -&gt; WireGuard Mesh -&gt; Home Nodes ✅</li><li>Internet User -&gt; OCI VPS Entry (FRP 443) -&gt; Home Nodes' Services (Nginx) ✅</li></ul></li></ul><p>이 과정에서 VCN 기본 대역(<code>10.0.0.0/24</code>)과의 충돌을 피하기 위해 VPN 내부망은 <code>10.0.1.0/24</code>로 설정했다. ( 뭣도 모르고 VCN Network 대역을 설정했다가 VPN 망 대역과 충돌하여 애좀 먹었었다..)</p><h4 id="22-%EC%A0%91%EC%86%8D-%EC%9D%B8%ED%94%84%EB%9D%BC-%ED%91%9C%EC%A4%80%ED%99%94">2.2. 접속 인프라 표준화</h4><p>관리 복잡도를 낮추기 위해 접속 방식을 통일했다.</p><ol><li>전체 노드에 대해 Password Login 비활성화 및 Key 기반 인증 적용.</li><li>복잡하게 꼬여있던 기존 SSH 키들을 폐기하고 마스터 키 생성 및 재배포.</li><li>~/.ssh/config 파일 설정으로 SSH ProxyJump 및 서버 별칭 구성</li></ol><p><strong>결과 : 클라이언트는 통일된 한 종류의 마스터키 하나만 가지고 있다면 비밀번호 입력도 없이 간편히 "ssh &lt;ServerName (ex. arm)&gt;" 만 입력해도 각 노드에 접근 가능하며, 비밀번호 기반 인증은 비활성화하여 보안 또한 강력해졌다.</strong></p><p><strong>또한, 인터넷에 노출되어있는 클라우드가 만에 하나 뚫려도, 키 파일은 전부 클라이언트 로컬 파일에만 존재하고 이들 접속은 Bastion 허브 서버를 통해 ProxyJump SSH 접속하기 때문에 안전할 것이다.</strong></p><hr><h3 id="3-architecture-%EB%B3%B4%EC%95%88%EA%B3%BC-%EA%B0%80%EC%9A%A9%EC%84%B1%EC%9D%84-%EB%B6%84%EB%A6%AC%ED%95%9C-%EC%B2%A0%EC%98%B9%EC%84%B1-%EA%B5%AC%EC%A1%B0">3. Architecture: 보안과 가용성을 분리한 '철옹성' 구조</h3><p>이번 마이그레이션의 핵심은 <strong>일반 사용자(Service) 트래픽</strong>과 <strong>관리자(Admin) 트래픽</strong>의 물리적/논리적 분리다.</p><h4 id="%F0%9F%8F%9B%EF%B8%8F-dual-path-hybrid-architecture">🏛️ Dual-Path Hybrid Architecture</h4><p></p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://yeoun.org/content/images/2026/02/image-3.png" class="kg-image" alt="" loading="lazy" width="1949" height="1085"><figcaption><span style="white-space: pre-wrap;">홈랩 사설망 아키텍쳐 (2026.02.07 기준)</span></figcaption></figure><ul><li><strong>Path A: The Admin Road (Stealth Mode)</strong><ul><li><strong>진입점:</strong> OCI ARM 인스턴스</li><li><strong>프로토콜:</strong> WireGuard UDP (단 하나의 포트만 개방)</li><li><strong>특징:</strong> 외부에서는 서버의 존재 자체를 알 수 없음(Ping Drop). 인증된 관리자만 Mesh Network를 통해 N4200/N4000 등 내부 서버의 SSH 및 DB에 접근 가능.</li></ul></li><li><strong>Path B: The Public Road (Service Mode)</strong><ul><li><strong>진입점:</strong> OCI VPS (Nginx Proxy Manager)</li><li><strong>프로토콜:</strong> HTTP/HTTPS</li><li><strong>특징:</strong> 웹 서비스 요청만 필터링하여 FRP 터널을 통해 집 내부의 워커 노드(N4200)로 전달. 사용자는 절대 내부 시스템(SSH/DB)에 접근할 수 없음.</li></ul></li><li><strong>각 노드 들은 물리적으로 떨어져 있어도 Wireguard 망을 통해 서로 안전하게 통신할 수 있다.</strong></li></ul><p></p><p></p><hr><h3 id="4-why-next-step-%EC%9D%B4-%EA%B5%AC%EC%A1%B0%EA%B0%80-%ED%95%84%EC%9A%94%ED%95%9C-%EC%9D%B4%EC%9C%A0">4. Why &amp; Next Step: 이 구조가 필요한 이유</h3><p>이 아키텍처는 단순히 "연결되었다"에 그치지 않고, 미래의 확장성을 고려한 설계다.</p><ol><li><strong>CI/CD &amp; Orchestration:</strong>외부 접속이 차단된 안전한 내부망(Mesh Network) 위에서 Docker Swarm 혹은 K3s 클러스터를 구성하여 자동화된 배포 파이프라인을 구축할 예정이다.</li><li><strong>Scalability (To the A100):</strong>향후 학부 연구생 활동 등을 통해 고성능 GPU 서버(A100 등) 권한을 얻게 될 경우, 물리적 위치와 상관없이 WireGuard 클라이언트 하나만 설치하면 즉시 내 사설 클러스터의 '고성능 연산 노드'로 편입시킬 수 있다.</li></ol><p>이제 하드웨어와 네트워크라는<strong> </strong>'도로'는 깔렸다. 다음은 이 도로 위를 달릴 '자동차(Application)'들을 컨테이너에 실어 N4000에서 N4200으로 이주시키는 작업을 시작한다.</p><hr><p><strong>[Next Action]</strong></p><ul><li>Docker Swarm 클러스터 이니셜라이징</li><li>20개 가량의 Legacy 서비스(N4000) 마이그레이션 및 분산 아키텍쳐 완성</li><li>Ansible 통한 IaC 구조 구성</li></ul>
