<h2 id="1-%EC%84%9C%EB%A1%A0-%EC%82%AC%EC%9A%A9%EC%9E%90-%EA%B2%BD%ED%97%98ux%EA%B3%BC-%EC%9D%B8%ED%94%84%EB%9D%BC-%EC%84%A4%EA%B3%84">1. 서론: 사용자 경험(UX)과 인프라 설계</h2><p>단순히 서버를 여는 것과 서비스를 구축하는 것의 차이는 <strong>사용자 편의성</strong>에 있다고 생각합니다. 인터넷을 찾아보면 vpn(wireguard, tailscale), VLAN(하마치) 등 멀리 떨어진 플레이어에게 서버를 열 수 있는 방법은 많습니다. 다만 클라이언트들의 설치가 필요하거나 더 깊은 설정이 필요하고 친구가  복잡한 <code>IPv4:Port</code> 주소조차 필요없이 외우기 쉬운 도메인 하나로 언제 어디서든 접속할 수 있는 환경을 만드는 것이 목표였습니다.</p><p>제가 보유한 최상의 자원인 <strong>Ryzen 5 9600X(DDR5 32GB, RTX 5060)</strong> 사양의 메인 PC를 서버 노드로 활용하고자 했습니다. 압도적인 성능을 제공하지만, 운영 환경을 고려하여 몇 가지 문제를 생각해볼 수 있었습니다.</p><h3 id="1-%EB%B3%B4%EC%95%88-%EC%9C%84%ED%98%91-security-risk">1. 보안 위협 (Security Risk)</h3><p>마인크래프트의 기본 포트인 <strong>TCP 25565</strong>를 개방하기 위해 공유기에서 포트포워딩(Port Forwarding)을 수행하는 순간, 우리 집의 <strong>공인 IP가 인터넷에 노출</strong>됩니다. 이는 <em>Shodan</em>과 같은 악성 스캐너의 직접적인 타겟이 되어 보안 취약점 공격에 무방비로 노출될 위험이 큽니다.</p><h3 id="2-%EA%B0%80%EC%9A%A9%EC%84%B1-%EC%A0%80%ED%95%98-availability-issues">2. 가용성 저하 (Availability Issues)</h3><p>악성 봇이나 DDoS 공격이 발생할 경우, 트래픽 과부하로 인해 집안 전체의 인터넷 네트워크를 관리하는 <strong>공유기가 다운</strong>될 수 있습니다. 이는 개인의 프로젝트 실패를 넘어 가족 공동의 인터넷 인프라가 마비되는 결과를 초래합니다.</p><hr><h3 id="%EC%99%9C-nginx-proxy-managernpm%EB%A5%BC-%EC%93%B0%EC%A7%80-%EB%AA%BB%ED%95%98%EB%82%98">왜 Nginx Proxy Manager(NPM)를 쓰지 못하나?</h3><p>이미 웹 서비스 배포용으로 사용 중인 NPM을 활용하려 했으나 명확한 한계가 있었습니다. Nginx는 기본적으로 <strong>L7(Application Layer)</strong> 정보를 기반으로 작동하는 역프록시(Reverse Proxy)입니다. 하지만 마인크래프트 패킷은 <strong>TCP(L4)</strong> 통신이므로, HTTP 헤더 정보가 없는 순수 TCP 패킷을 NPM만으로 처리하기에는 제약이 컸습니다.</p><h3 id="%EC%82%AC%EC%9A%A9%EC%9E%90-%EA%B2%BD%ED%97%98ux-vs-%EA%B8%B0%EC%88%A0%EC%A0%81-%ED%8E%B8%EC%9D%98%EC%84%B1">"사용자 경험(UX) vs 기술적 편의성"</h3><p><em>Cloudflare Tunnel</em> 도입도 고려했지만, 외부 접속자(친구들)가 별도의 클라이언트 프로그램을 설치해야 한다는 점이 큰 장벽이었습니다. 저는 **"주소창에 도메인 하나만 치고 들어온다"**는 UX를 포기할 수 없었습니다.</p><h3 id="%EC%B5%9C%EC%A2%85-%EC%84%A0%ED%83%9D-fast-reverse-proxy-frp">최종 선택: Fast Reverse Proxy (FRP)</h3><p>이러한 제약 사항을 모두 해결하기 위해 Go 언어 기반의 **Fast Reverse Proxy (FRP)**를 도입했습니다.</p><ul><li><strong>Public IP 은닉:</strong> OCI VPS를 Entry Point로 사용하여 실제 서버의 IP 주소를 숨깁니다.</li><li><strong>L4 Tunneling:</strong> TCP 패킷을 암호화된 터널로 안전하게 중계합니다.</li><li><strong>No Client-side Install:</strong> 접속자는 별도의 프로그램 없이 도메인 주소만으로 접속이 가능합니다.</li></ul><p></p><h2 id="2-oci%EC%99%80-edge-node%EC%9D%98-%EA%B2%B0%ED%95%A9">2. <p><strong>1.  Request:</strong> 외부 클라이언트가 도메인을 통해 접속을 시도합니다.</p><ol start="2"><li><strong>Public Entry:</strong> 패킷은 먼저 **OCI VPS(FRPS)**에 도착합니다. 이때 집의 실제 공인 IP는 철저히 은닉됩니다.</li><li><strong>Secure Tunneling:</strong> VPS의 FRP 서버는 암호화된 <strong>Fast Reverse Proxy Tunnel</strong>을 통해 집 내부의 **N4000(Control Node)**으로 패킷을 토스합니다.</li><li><strong>Local Routing:</strong> N4000의 FRP Client가 패킷을 수신하고, 내부망(LAN)을 통해 실제 게임 서버가 구동 중인 **PC(Ryzen 9600X)**로 트래픽을 라우팅합니다.</li><li><strong>Response:</strong> 게임 데이터는 역순으로 터널을 타고 클라이언트에게 안전하게 전달됩니다.</li></ol><h2 id="%ED%95%9C%EA%B3%84%EC%A0%90">한계점</h2><p>초기에는 메인 작업용 PC(9600X)를 연산 노드로 활용하고 OCI를 게이트웨이로만 사용하는 하이브리드 방식을 채택했으나, 실제 운영 시 다음과 같은 고질적인 문제들에 직면했습니다.</p><h3 id="1-%EC%84%9C%EB%B9%84%EC%8A%A4-%EA%B0%80%EC%9A%A9%EC%84%B1%EC%9D%98-%EC%A0%9C%EC%95%BD-247-%EC%9A%B4%EC%98%81-%EB%B6%88%EA%B0%80">1.  서비스 가용성의 제약: 24/7 운영 불가</h3><ul><li><strong>문제:</strong> 메인 PC는 개인 작업 및 휴식을 위해 수시로 재부팅되거나 종료됩니다.</li><li><strong>영향:</strong> 서버를 24시간 안정적으로 호스팅할 수 없었으며, 사용자가 접속하고 싶을 때 서버가 닫혀 있는 상황이 빈번하게 발생하여 서비스 신뢰도가 하락했습니다.</li></ul><h3 id="2-%EB%A1%9C%EC%BB%AC-%EB%A6%AC%EC%86%8C%EC%8A%A4-%EC%9E%A0%EC%8B%9D-jvm-%EB%A9%94%EB%AA%A8%EB%A6%AC-%EC%A0%90%EC%9C%A0-%EB%AC%B8%EC%A0%9C">2. 로컬 리소스 잠식: "JVM 메모리 점유 문제"</h3><ul><li><strong>문제:</strong> 마인크래프트 서버(Java) 특성상 안정적인 구동을 위해 **8GB 이상의 RAM(Xmx8G)**을 사전에 할당(Reservation)해야 했습니다.</li><li><strong>영향:</strong> 32GB의 램 중 25% 이상이 상시 점유되어, 메인 PC에서 고사양 게임이나 무거운 작업을 병행할 때 시스템 병목 현상이 발생했습니다.</li></ul><h2 id="3-oci-cloud-native%EB%A1%9C%EC%9D%98-%EC%A0%84%ED%99%98-%EC%B5%9C%EC%A0%81%ED%99%94-%EB%B0%8F-%EC%9E%90%EB%8F%99%ED%99%94">3. OCI Cloud Native로의 전환 (최적화 및 자동화)</h2><p>로컬 서버 운영의 한계를 극복하기 위해 모든 컴퓨팅 자원을 **OCI(Oracle Cloud Infrastructure)**로 이전하며, 하이브리드 클라우드 구조를 클라우드 네이티브 환경으로 고도화했습니다.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://yeoun.org/content/images/2026/01/image-1.png" class="kg-image" alt="" loading="lazy" width="1005" height="567"><figcaption><span style="white-space: pre-wrap;">구조가 오히려 단순해졌다.</span></figcaption></figure><h3 id="1-%EC%8B%9C%EC%8A%A4%ED%85%9C-%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98-%EB%B3%80%ED%99%94-before-after">1. 시스템 아키텍처 변화 (Before &amp; After)</h3><ul><li><strong>이전 (Hybrid):</strong> <code>OCI VPS</code> → <code>FRP Tunnel</code> → <code>Home Network</code> → <code>9600X</code><ul><li>물리적 거리에 따른 <strong>Network Latency</strong> 발생 및 집 인터넷 환경에 따른 가용성 의존도 높음.</li></ul></li><li><strong>현재 (Full Cloud):</strong> <code>OCI VPS (AMD)</code> ↔ <code>OCI ARM Node (24GB RAM)</code><ul><li>오라클 내부 **고속 백본망(VCN)**을 활용하여 데이터 경로를 단축하고 응답 속도를 극대화함.</li></ul></li></ul><h3 id="2-%EA%B8%B0%EC%88%A0%EC%A0%81-%ED%95%B5%EC%8B%AC-%EC%9D%B8%EC%82%AC%EC%9D%B4%ED%8A%B8">2. 기술적 핵심 인사이트</h3><ul><li><strong>리소스 극대화:</strong> Always Free 티어의 <strong>ARM Ampere A1</strong> 노드를 활용, <strong>4 OCPU와 24GB RAM</strong>을 할당하여 메모리 집약적인 Java 애플리케이션(Minecraft)을 비용 0원에 안정적으로 구동.</li><li><strong>보안 통일성:</strong> VCN 내에서도 **FRP(L4 Tunneling)**를 유지하여, 추후 홈 서버(9600X)로의 확장성을 열어둠과 동시에 보안 관문을 단일화함.</li><li><strong>운영 자동화 (L7):</strong> 서비스 로그를 실시간 파싱하여 <strong>Discord Webhook API</strong>와 연동. 인프라 상태를 모바일로 즉시 확인 가능한 모니터링 체계 구축.</li></ul><h3 id="3-why-oci-arm-node">3. Why OCI ARM Node?</h3><ul><li><strong>가용성 확보:</strong> 99.9% 가동률을 보장하는 OCI 클라우드 인프라로 24시간 무중단 서비스 실현.</li><li><strong>로컬 자원 해방:</strong> 메인 PC의 RAM 8GB를 다시 온전한 작업 용도로 회복.</li><li><strong>비용 제로 최적화:</strong> Always Free 티어의 <strong>24GB RAM</strong>을 활용하여, 내 컴퓨터보다 더 넉넉한 자원을 비용 지출 없이 서버에 할당.</li></ul><h3 id="4-%EC%99%9C-vcn%EC%9D%B4-%EC%95%84%EB%8B%8C-frp%EC%9D%B8%EA%B0%80">4. 왜 VCN이 아닌 FRP인가?</h3><p>OCI 내부 인스턴스 간(AMD ↔ ARM) 통신은 일반적인 VCN(Virtual Cloud Network) 설정만으로도 가능합니다. 하지만 저는 모든 구간에 **FRP(Fast Reverse Proxy)**를 적용했습니다. </p><ol><li><strong>보안 정책의 통일성:</strong> 집(Home-Lab) 서버는 보안상 공인 IP 은닉을 위해 반드시 FRP 터널이 필요합니다. 클라우드 노드 간 통신에도 동일한 아키텍처를 적용하여 <strong>관리 포인트의 통일성</strong>을 확보했습니다.</li><li><strong>공격 표면 최소화:</strong> 모든 외부 트래픽은 VPS(AMD)라는 단일 관문을 통해서만 들어옵니다. 이를 통해 실제 연산 노드(ARM)의 직접 노출을 차단했습니다.</li><li><strong>Zero-day 대응력:</strong> 마인크래프트 서버 자체에 Zero-day 취약점이 발생하더라도, Docker 기반의 FRP 격리와 VPS 단계의 <strong>Fail2Ban</strong>을 통해 내부망 침투 시도를 입구에서 차단할 수 있는 계층적 방어 체계를 구축했습니다.</li></ol><h2 id="4-troubleshoot-split-dns-issue%EC%99%80-l3-%EA%B3%84%EC%B8%B5%EC%9D%98-%EC%9D%B4%ED%95%B4">4. Troubleshoot: Split DNS Issue와 L3 계층의 이해</h2><p>인프라 구축 중 발생했던 문제로 내부망 접속 불가 현상이 있었습니다.</p><ul><li><strong>증상:</strong> 외부에서는 도메인으로 잘 접속되는데, 집 내부망에서는 접속이 실패함.</li><li><strong>원인 분석:</strong> 집 내부 DNS 서버인 <strong>AdGuard Home</strong>의 DNS Rewrite 중 와일드카드 설정 때문이었습니다. 내부 사용자가 도메인을 질의할 때, 패킷이 OCI 공인 IP로 나가지 못하고 내부 사설 IP(그것도 완전히 다른 노드)로 유도되는 경로 오류가 발생했습니다.</li><li><strong>해결:</strong> AdGuard Home에서 특정 도메인에 대한 DNS Rewrite 예외 설정을 적용하여 해결했습니다. 이 과정을 통해 L3 계층에서 네임 서버가 패킷 경로에 미치는 영향을 다시 생각해보았습니다.</li></ul>
